# Phase 28 KNOWLEDGE（MF 実機調査 2026-07-05）

MoneyForward MCP（`mfc_ca_getAccounts` / `mfc_ca_getJournals`）で確認した実挙動・制約。
※ Git 追跡対象。カード名・補助科目名・銀行名・事業者名・取引先コード・金額等の実値は書かない（PII）。

## 連携カードは MF 上の登録先勘定科目で記帳が変わる

同じ「個人用クレジットカード連携」でも、MF 上でその連携口座をどの勘定科目配下に
登録するかで、課金候補の貸方が変わる。

- **未払金配下に登録**：課金候補は `通信費 / 未払金/<カード補助科目>` で入る。
  → 個人資金で払う場合、別途 `未払金→事業主借` の振替が必要（2段階）。
- **事業主借配下に登録**：課金候補は `通信費 / 事業主借/<カード補助科目>` で直接入る。
  → 引き落とし振替は不要（1段階）。

個人用カードを「未払金を経由せず事業主借直接」にしたい場合は、
**MF の連携口座設定でその口座を事業主借配下に登録する**のが正攻法（登録後の課金から反映）。
`/register` の未仕訳フォーム側で貸方勘定を差し替えるのではなく、連携口座の設定で決まる。

## 事業主借にもカード/口座別の補助科目が付く

`事業主借`（OWNERS_CAPITAL）配下にカード・口座別の補助科目が複数存在する。
個人用カードの課金・振替の貸方は `事業主借/<カード別補助科目>` になる（補助科目なしではない）。
どのカードにどの補助科目を割り当てるかはユーザー設定であり、実値は `local.yml` の
`credit_cards[].owner_sub_account` に持つ（追跡ファイルには書かない）。

## `mfc_ca_getConnectedAccounts` は空配列を返すことがある

カードが実際には連携済み（未払金/事業主借に連携由来の補助科目が存在する）でも、
`getConnectedAccounts` は `{"connected_accounts":[]}` を返した。連携カードの特定には
このAPIは当てにできず、`getAccounts` の補助科目や `getJournals` の実仕訳から推定する。

## 勘定科目・補助科目 ID はURLエンコード済み文字列

`getAccounts` が返す `id` / `sub_accounts[].id` は `%3D%3D` 等を含むURLエンコード済み文字列。
`getJournals` の `account_id` 等にそのまま渡して使える（MCP 指示どおり ID は無加工で使用）。
ID 実値はセッション・事業者依存のため、必要時に `getAccounts` で都度解決する（ハードコードしない）。
