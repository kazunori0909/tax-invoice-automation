# Phase 28: クレジットカード種別（個人用＝事業主借直接／事業用＝未払金）対応

※ 本ファイルは Git 追跡対象。カード名・補助科目名・銀行名等の実値は書かない（PII。実値は `local.yml`）。

## 課題

カード決済サービスの課金の記帳が、カードごとにバラバラで仕様と実態も食い違っていた。
MF 実データ（2026-07-05 調査）で判明した現状：

| カード（個人用） | MF 連携の登録先 | 課金の仕訳 | 引き落とし |
|---|---|---|---|
| カードA（現 anthropic/google-ai/moneyforward の決済） | **未払金** | 通信費 / 未払金（連携自動） | 26日に別仕訳 `未払金→事業主借`（credit-payment 月次） |
| カードB（現 xserver/moomoo の決済） | **事業主借** | 通信費 / 事業主借（**1仕訳で直接**） | 不要 |

- カードB は連携口座が MF 上で「事業主借」配下に登録されており、課金が最初から事業主借直接。
- カードA だけ連携口座が「未払金」配下で、2段階（課金＝未払金 → 引き落とし振替）になっていた。
- `services/*.yml` の `journal_rule` は全カードで `credit_account: 未払金` と書かれており、
  カードB（実態は事業主借）と食い違っていた。

## 方針（ユーザー承認済み・2026-07-05）

**個人用カードは「事業主借直接」に統一する。** カードA もカードB と同じ方式へ寄せる。

| | 個人用カード（現行の全カード） | 事業用カード（将来用・現存せず） |
|---|---|---|
| MF 連携の登録先 | 事業主借（カード別補助科目） | 未払金 |
| 課金の仕訳 | **通信費 / 事業主借 / <カード補助科目>**（1仕訳） | 通信費 / 未払金 / <カード補助科目> |
| 引き落とし | なし（個人資金で完結） | 未払金 / 普通預金（引落口座も連携なら MF 自動、未連携なら credit-payment） |
| `/register` | 連携＋自動仕訳ルールの結果を確認（貸方＝事業主借/補助科目） | 同（貸方＝未払金/補助科目） |
| `/credit-payment` | 対象外 | 対象（未払金の消し込み確認） |

### 設定（`credit_cards[]`）

- `type: personal | business`（省略時 `business`）。
- **個人用**：`name`（＝`services.<svc>.card` と名前一致）／`owner_sub_account`（事業主借の補助科目）。
- **事業用**：`name`／`closing`/`payment_day`（`card_account`＝未払金・`bank_account`＝普通預金は `type` から導出。
  `card_sub_account`/`bank_sub_account` は任意＝引き落とし口座が未連携で手動登録する場合のみ。連携時は補助科目も自動仕訳から取得）。
- 紐づけキー＝名前一致：`services.<svc>.card` == `credit_cards[].name`。
- 名称は MF の実名称に合わせる（実値は `local.yml` のみ。追跡ファイルはプレースホルダー）。

### 貸方はカード依存（サービス直書きしない）

`services/*.yml` の `journal_rule` は貸方を直書きせず、`services.<svc>.card` → `local.yml` の
`credit_cards[]` の `type` で貸方を決定する（register が解決）。カードを変えても矛盾しないように
するため（同じサービスがカードを乗り換える場合に対応）。フラグ（`credit_by_linked_card`）は不要と判断し置かない。

### スキル

- **register**：課金は連携＋自動仕訳ルールで登録され、register は結果を確認するだけ（現行どおり）。
  個人用カードの期待貸方＝`事業主借/owner_sub_account`。MF がまだ未払金で入っている場合は
  「MF 連携口座の登録先を事業主借に変更してください」とユーザーに通知する（＝カードA の移行検知）。
  未払金→事業主借の振替仕訳は作らない（直接記帳のため不要）。
- **credit-payment**：`type: business`（未払金型）専用に役割を絞る。個人用は対象外。
  現状は事業用カードが存在しないため、対象カードなしでレポートするだけ。貸方は普通預金（引落口座）。

## ユーザー側の移行作業（MF UI・コード対象外）

1. カードA の**連携口座の登録先を「未払金」→「事業主借（カード別補助科目）」に変更**。
2. 変更前に未払金へ入っていた既存残高を手動で事業主借へ修正（ユーザーが実施予定）。
3. 移行後、`/register anthropic` 等で貸方が事業主借/該当補助科目になることを確認。

## 決定事項ログ

### 2026-07-05: 個人用カードを事業主借直接に統一

- **実データ調査**：カードB は既に事業主借直接、カードA のみ未払金2段階だった（上表）。
- **決定**：個人用は事業主借直接に統一。カードA もカードB 方式へ寄せる（ユーザー承認）。
  当初案（register で未払金→事業主借の振替を追加）は破棄。振替・未払金を使わない直接記帳が
  よりクリーンで、カードB の既存運用とも一致するため。
- **カードA の事業主借補助科目はユーザー選択（カード名ベース）**。実値は `local.yml` の `owner_sub_account`。
- **貸方はカード依存**（`services.card` → `credit_cards[].type`）。サービス yml に貸方勘定を直書きしない。
- **type から自明な勘定科目は config に書かない**（business の `card_account`＝未払金・`bank_account`＝普通預金は skill 側で導出）。フラグ `credit_by_linked_card` も置かない。
- **事業用カードの補助科目も config 任意**：引き落とし口座も連携済みなら課金・引き落としとも MF 自動生成で、
  補助科目は自動仕訳から動的取得できるため config 記述不要（極論エントリ自体ほぼ不要。confirm するなら name/type/closing/payment_day のみ）。
  `card_sub_account`/`bank_sub_account` は引き落とし未連携で手動登録する場合のみ指定。
- **紐づけキー＝名前一致**（`services.card == credit_cards.name`）。
- **現存カードは全て個人用**。事業用（未払金型）カードは存在しないが、`business` type と
  credit-payment は将来用に仕様・雛形として残す。
- **移行は MF UI 作業**：カードA 連携口座の登録先変更＋既存未払金残高の修正はユーザーが実施。

### 2026-07-06: journal_rule のスリム化（ユーザー承認）

- `services/*.yml` の `journal_rule` から以下を削除し、**借方勘定科目＋貸方カード依存コメントのみ**の最小形に統一：
  - `tax`：ユーザーの課税状態（免税/課税事業者）依存の個人値であり、実際の税区分は適格請求書・MF 自動仕訳が決める。git 管理の汎用仕様に書く値ではない
  - `trade_partner`（journal_rule 内）：トップレベル `trade_partner:` と完全重複
  - `transaction_date` / `billing_cycle` / `evidence` / `yearly_note` / `domains_note`：同ファイル内の `date_rule`・`billing_modes`・`memo_source`・冒頭コメントと重複
  - `amount_source` / `setup_note`：全サービス共通の手順のため、register/SKILL.md の「全サービス共通ルール」へ移動（データ層→手順層）
  - `status`（確定/暫定/TBD）：ワークフロー状態。未確定事項は既存の「★未確認」コメント規約とフェーズ記録で管理
- register/SKILL.md Step 5 に共通ルールを明記：金額＝カード明細の円決済額／税区分＝MF 自動設定のまま変更しない／初回のみ手動設定
- service-new/SKILL.md の yml 雛形も新しい最小形に更新（旧フィールドを生成しない）
