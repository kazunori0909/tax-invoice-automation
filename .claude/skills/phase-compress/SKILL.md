# フェーズ圧縮

CLAUDE.md の運用ルールどおり、**このスキルはユーザーが明示的に圧縮を指示したときのみ実行する**
（Claude Code の自己判断で phases.md を更新しない）。

## 引数

- **フェーズ番号**（任意）: 圧縮対象の指定（例: `16 17 20`・`21まで`）。
  未指定の場合は完了済みフェーズ全部を候補にする。

## 手順

### Step 1: 対象の確定

1. `docs/phase*/` のフォルダを列挙する。
2. 各フェーズの完了判定: `TASKS.md` のチェックボックスが**全て `[x]`**、かつ
   `docs/README.md` のテーブルで「進行中」表記でないこと。
3. 未完了・進行中のフェーズは対象から**除外**する。引数で明示指定されていた場合は、
   未完了である旨を伝えてユーザーの判断を仰ぐ。
4. 対象候補の一覧（番号・タイトル・完了状態）を提示し、圧縮対象を確定する。

### Step 2: 統合テキストの作成

各対象フェーズのフォルダ内**全ファイル**（PLAN.md・TASKS.md・KNOWLEDGE.md・`decision-*.md` 等）を読み、
`docs/phases.md` の既存スタイルに凝縮する:

```markdown
## Phase <N>: <タイトル>

<現行アーキテクチャに効いている決定・確定仕様を、理由込みで2〜5行程度の地の文で>
```

- **残すもの**: **現在も有効な**決定事項（何を選び、なぜか）／確定した仕様／実機調査で判明した
  制約・挙動のうち現行仕様の理解に必要なもの
- **落とすもの**:
  - タスク一覧・スコープ・完了条件などの作業管理情報
  - 案A/案B比較・検討過程・バグ修正の経緯など、結論に至る過程の詳細（結論と理由の一言だけ残す）
  - 操作手順そのもの（`.claude/skills/**/SKILL.md`・`download/references/<svc>.md` 等の手順層に
    書くべき内容。phases.md には「どこに書いてあるか」の一言のみ残す）
  - **後続フェーズで上書き・撤回された決定**（Phase 番号は欠番にしてよい。現行コード・設定を
    grep 等で確認し、当該フェーズの決定が今も実在するか必ず裏取りする）
  - 日付（`YYYY-MM-DD` 等の厳密な日付は付けない。フェーズ番号が順序を表すため不要）
- **PII マスク**: 過去フェーズの docs には実値（金額・ドメイン・コード等）が残っていることがある。
  phases.md へは転記時にプレースホルダー化する（SECURITY.md §9）。
- 対象フェーズの決定が既に他の後続フェーズ（未圧縮含む）で上書きされていた場合は、
  `docs/README.md` のフェーズ記録テーブルの「要点」列にその旨を一言添える
  （例:「後に◯◯へ置き換え。詳細は phases.md に非掲載」）。

### Step 3: ユーザー承認

統合テキストの全文と削除予定フォルダの一覧を提示し、承認を得る。
**承認を得るまで書き込み・削除を行わない。**

### Step 4: 反映

1. `docs/phases.md` にフェーズ番号順で追記する。
2. phases.md 冒頭の対象範囲表記（「Phase 1–15」等）を実態に合わせて更新する。
3. `docs/README.md` を更新する:
   - テーブルの該当行は**残し**、「記録」列のリンクをフォルダ → `[phases.md](phases.md)` に付け替える
   - 「Phase 1–15 は phases.md に統合済み」等の範囲記述を実態に合わせて更新する
4. 対象フォルダを削除する（`git rm -r docs/phase<N>-<slug>/`。git 履歴には残るため復元可能）。

### Step 5: 個人情報チェック

phases.md・docs/README.md の追記・変更箇所に個人情報が混入していないか最終確認する。
